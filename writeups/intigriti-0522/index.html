/ Analysis

An element of the DOM have its innerHTML attribute set to a dynamically retrieved value from the hardcoded pages associative array, based on a key that we can control through the page query parameter.

If the key exists and a value is retrieved from the array, it is sanitized using the filterXSS method from jsxss before being injected into the DOM.

So, if we could somehow:

  1. Inject a key-value element in the array (or as-if üëÄ?)
  2. Bypass the sanitizing applied on the injected value

‚û° Then we would have reflected XSS !

/ Part 0x1: the injection

After some digging and having not that much of custom code to play with, my attention turned to the javascript library being used.

The loaded jquery-query-object@2.2.3 seems to be known vulnerable to a client-side prototype pollution.

A simple test confirms it:

?__proto__[erk3]=test&page=1

But Javascript is made so that if the key we provide for looking up a value in an associative array does not exist, it will return any attribute from the prototype of the array object having the same name.

The trick is then to leverage this mechanism so we can set any key (as-if) of the pages array to the value we want !

?__proto__[erk3]=<script>alert('XSS?')</script>&page=erk3

At this point we can inject arbitrary HTML/JS code but still not achieve XSS due to the sanitizing.

/ Part 0x2: the bypass

Having only to bypass the filterXSS now, a quick search on the Internet can lead to one of the great PortSwigger cheatsheet:

Conveniently for us we already have the necessary prototype pollution, making it as simple as the following to whitelist the src and onerror attributes to the <img/> tag.

?__proto__[whiteList][img][]=onerror&__proto__[whiteList][img][]=src
// Allowing payload like: <img src onerror=alert(1) />

And we should be done!
